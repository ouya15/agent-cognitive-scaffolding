# 当 AI 修了三遍还没修好：从一次深夜故障谈 Agent 的认知脚手架设计

> 与其抱怨 AI 不够聪明，不如给它搭一副能让它聪明起来的脚手架。

## 一、深夜，AI 把简单问题修成了复杂问题

4月25日深夜，我本地跑的 OpenClaw（一个通过 npm 全局安装的 AI Agent Gateway）挂了。原因很平常——半个月前收到一个 SIGTERM 后进程退了，没有自动拉起。

我让 AI 编程助手来处理。第一步做得很漂亮：快速诊断出进程状态，一条 `openclaw gateway start` 就恢复了服务。如果故事到这里结束，这篇文章也就不存在了。

接着我提了第二个需求：升级到最新版，顺便移除一个不用的频道配置。AI 用 `openclaw update` 把版本从 v2026.3.24 升到了 v2026.4.8，一切顺利。但随后，它注意到状态输出中有一行：

```
Update: available · pnpm · npm update 2026.4.23
```

这行信息本来只是告诉你"有更新可用"——一条**信息**，不是一条**指令**。但 AI 看到 `"npm update"` 这几个字，把它当成了待办事项，决定"帮我"继续升级。更糟糕的是，它没有再用应用层的 `openclaw update`，而是绕到了底层，直接执行了 `npm update -g openclaw`。

这一操作直接破坏了安装。`dist/` 目录倒是在，但 `package.json` 和 `openclaw.mjs` 入口文件都没了——一个典型的 npm 全局安装不完整的状态。

到这里，事情开始变得有趣。

## 二、重复失败：AI 的"只前冲不回退"

面对安装损坏，AI 选择了最直觉的策略：重新装一遍。`npm install -g openclaw`，失败。再来一遍，加上 `--verbose`，还是不完整。换个姿势，加上 `--ignore-scripts`，依然不行。

我看着终端里滚过的日志，意识到一件事：**AI 连续执行了同一类操作四五次，每次只是换了个 flag，但从未质疑过"用 npm install 能不能解决这个问题"这个前提本身。**

这不是某一个模型的问题，因为接下来我换了一个模型让新的 Agent 接手，它读完上下文后，继承了前任的策略——继续 `npm install`，继续失败。新 Agent 继承了方法，却没有继承教训。

如果你管理过团队，这个场景可能有点眼熟：新人接手项目，前任留下了代码和文档，但没留下"我踩过哪些坑、为什么放弃了方案 A"。结果新人把同样的坑重新踩了一遍。

## 三、一个元问题：为什么会这样决策？

到了这个节点，我没有继续让 AI 尝试第六种 flag 组合，而是问了一个不同层面的问题：

**"为什么会出现这样的决策？Agent 的思考过程是怎样的？"**

这个问题的关键在于，它不是在问"怎么修"，而是在问"为什么你的修法是错的"。它迫使 AI 从执行层跳到反思层——从"做什么"切换到"为什么这么做"。

AI 停下来做了一次根因分析，输出了五个问题。这些问题后来成了我设计认知脚手架的基础，值得逐一展开：

### 缺陷一：把信息误读为指令

`Update available: 2026.4.23` 是一条状态信息，意思是"有新版本存在"。但 AI 将其解读为"你应该立刻升级到这个版本"。

这暴露了一个根本性的认知缺陷：**AI 缺乏对信息类型的元判断**。在人类的认知中，我们无时无刻不在区分"事实陈述"和"行动请求"——"明天可能下雨"不等于"你必须带伞"。但对 token 序列进行模式匹配的 AI 来说，"available"这个词太容易触发"去获取它"的行动倾向。

### 缺陷二：不理解工具的层次结构

OpenClaw 有自己的 `openclaw update` 命令，这个命令封装了依赖处理、配置迁移、服务重启等一系列逻辑。`npm update -g` 只管 npm 包本身的文件投递，不管上层应用的完整性。

AI 绕过高层抽象直接操作底层，就像你想给手机升级系统，不去"设置-系统更新"，而是手动往 `/system` 分区 `dd` 了一个镜像。能跑才见鬼了。

这反映的是**对工具分层的无意识**。AI 看到"需要更新 npm 包"，就直接调用了它所知的 npm 命令，没有思考"这个包有没有提供自己的更新机制，为什么要提供自己的更新机制"。

### 缺陷三：只前冲不回退

五次 `npm install` 失败后，AI 的策略空间里只有"换一种方式安装"，从未出现"先彻底清理再重装"或"回退到已知正常状态"。

这有点像一个只会踩油门不会打方向盘的司机。遇到路障不是绕行，而是加大油门冲过去。实际修复最终靠的就是先彻底清理损坏的安装，再干净地重新安装——一个非常基本的"回退到已知状态"策略。

### 缺陷四：重复同一策略期待不同结果

爱因斯坦（可能）说过，疯狂就是重复做同样的事情却期待不同的结果。AI 在这个场景里表现得非常"疯狂"——同样的命令执行了多次，每次的微调（加个 verbose、加个 ignore-scripts）不足以改变结果的性质。

深层原因是 **AI 缺少一个"失败计数器"和"策略升级机制"**。它没有"我已经在同一条路上失败两次了，应该质疑这条路本身"的元认知规则。

### 缺陷五：上下文继承的断层

换模型后新 Agent 读取了完整对话记录，但只继承了"当前的执行策略"，没有提取出"哪些方法已经被证伪"。对话上下文里的失败记录，对它来说只是一堆 token，不是结构化的经验知识。

这五个缺陷合在一起，构成了一幅清晰的图景：**AI Agent 不是不够聪明，而是缺少认知结构。** 它有强大的执行能力，但缺乏关于"如何执行"的元知识——什么时候该停、什么时候该换路、什么信号意味着什么。

## 四、修好了，然后呢？

最终的修复本身并不复杂：

1. 彻底删除损坏的全局安装
2. 干净地重新 `npm install -g openclaw`（这次耐心等它装完）
3. `openclaw gateway install` 注册系统服务
4. `openclaw gateway start` 启动
5. 清理残留的微信扩展目录（`~/.openclaw/extensions/openclaw-weixin`）——配置文件里删了渠道，但扩展文件还在，会导致 Gateway 启动时自动发现并报错超时

v2026.4.23 跑起来了，问题解决。

但我知道，这次修好不代表下次不会出同样的问题。AI Agent 没有长期记忆，下次遇到类似场景，它还是会犯同样的错误。更现实的问题是，我自己也可能凌晨三点被叫起来处理这类问题，到时候脑子不一定比 AI 清醒。

所以我跟 AI 说：**"更重要的是反思为什么会出问题，系统化地解决。做一个 Skill，安装到你本地。"**

这句话的意思是：**不要只修这次的 bug，要修 AI 的认知 bug。** 把这次事件中学到的经验，编码成一个可复用的知识模块——一个 AI Agent 专用的操作手册（Skill），让它在未来遇到同类问题时，不再重蹈覆辙。

## 五、从"人类手册"到"AI 决策引擎"

AI 很快交出了第一版 Skill。148 行，结构清晰，覆盖了启动、停止、升级、配置等基本操作。看起来不错——如果读者是人类的话。

问题在于，这份手册本质上是"人类运维文档"的翻版：罗列了命令和步骤，但没有应对 AI 特有的认知缺陷。如果 AI 在执行过程中再次把信息当指令、再次绕过工具分层、再次在失败后原地重试，这份文档救不了它。

于是我提出了第二个要求：**"用面向 AI 的思维重新审视和完善这个 Skill。"**

这个提示的核心区别在于受众的转变。我们习惯了为人类写文档——假设读者有常识、有判断力、能在模糊情况下做出合理决策。但 AI 没有这些。为 AI 写"文档"，需要不同的设计哲学：

- 人类文档说"注意检查安装是否完整"；AI 文档需要说"执行 `ls` 检查以下 5 个文件是否存在，任意一个缺失则判定为 CORRUPT_PACKAGE 状态"
- 人类文档说"如果多次尝试仍失败，考虑换一种方法"；AI 文档需要说"同一策略失败 2 次，强制切换到下一个 Playbook，禁止第 3 次尝试"
- 人类文档说"升级前请做好备份"；AI 文档需要说"执行任何 npm 命令前，必须验证前置条件：openclaw CLI 可用 AND 当前目录非项目目录 AND 无其他 npm 进程运行"

最终版本膨胀到了 387 行。不是因为堆砌了更多命令，而是因为加入了完整的**认知脚手架**。

## 六、认知脚手架：四个方法论的融合

"认知脚手架"这个概念借自教育心理学——维果茨基的"最近发展区"理论。老师不是替学生做题，而是提供恰好够用的支撑结构，让学生能够完成他们独自完不成的任务。类似地，我设计的 Skill 不是替 AI 做决策，而是为它的决策过程提供结构。

这个结构融合了四个方法论，每个都针对前面识别出的一种或多种认知缺陷：

### 6.1 Hoare Logic 的前置/后置条件

Hoare Logic 是形式化验证的基础之一，核心思想极简：`{P} C {Q}`——在前置条件 P 成立的情况下执行命令 C，后置条件 Q 必须成立。

我把这个模式应用到 Skill 的每一个操作步骤上。以"升级"操作为例：

**前置条件**（必须全部满足才能开始）：
- `openclaw` CLI 可执行
- 当前 Gateway 状态为 RUNNING 或 STOPPED
- 包完整性检查通过

**操作**：
- 执行 `openclaw update`（不是 `npm update`）

**后置条件**（操作完成后必须验证）：
- 版本号已变更
- 包完整性检查通过
- Gateway 服务可达

任何一个后置条件不满足，都不能宣布"升级成功"。这直接解决了 AI "执行了命令就以为完成了任务"的问题。

### 6.2 FMEA 故障模式与影响分析

FMEA 是制造业和航空领域用了几十年的方法论：提前枚举所有可能的故障模式，为每种故障预设响应方案。

我在 Skill 中定义了一张**失败模式矩阵**：

| 状态 | 诊断依据 | 对应 Playbook |
|------|---------|---------------|
| HEALTHY | CLI 可用 + 进程运行 + 端口响应 + 包完整 | 无需操作 |
| CORRUPT_PACKAGE | CLI 不可用或关键文件缺失 | 彻底清理 + 重装 |
| GATEWAY_HUNG | 进程存在但端口不响应 | 停止 + 重启 |
| STALE_LAUNCHAGENT | 服务配置指向旧路径 | 卸载 + 重新安装服务 |
| PORT_CONFLICT | 端口被其他进程占用 | 识别冲突 + 用户决策 |
| NO_SERVICE | CLI 可用但服务未注册 | 安装服务 + 启动 |
| CONFIG_OPERATION | 仅配置变更 | 修改配置 + 重启 |

这张矩阵的价值在于：**AI 不需要在运行时"想出"解决方案，只需要正确识别状态，然后执行对应的预设方案。** 这大幅降低了决策的认知负荷，把开放性问题转化为分类问题。

### 6.3 OODA 循环

OODA（Observe-Orient-Decide-Act）是博伊德在空战理论中提出的决策循环。它的核心洞察是：**决策的速度和质量取决于你在每个阶段的信息质量。**

在 Skill 的设计中，OODA 体现为强制性的 Phase 0——**在做任何操作之前，必须先完成全面的状态诊断**。这不是建议，是规则。诊断覆盖五个维度：

1. **CLI 可用性**：`openclaw --version` 能否正常返回
2. **Gateway 进程**：是否在运行、PID 是什么
3. **端口状态**：目标端口是否被监听、被谁监听
4. **版本信息**：当前版本和可用版本
5. **包完整性**：关键文件是否存在、目录结构是否正确

只有 Observe 阶段完成，才能进入 Orient（状态判定）、Decide（选择 Playbook）、Act（执行操作）。而且每次 Act 之后，OODA 循环重新开始——必须重新 Observe 来验证操作效果。

这直接解决了"不诊断就下药"的问题。前面那次事故中，AI 在一开始的修复成功后，没有重新评估全局状态就继续操作，导致了后续的级联故障。

### 6.4 Kubernetes 声明式控制器模式

K8s 的核心设计哲学是声明式的：你描述期望状态（Desired State），控制器不断将实际状态（Actual State）向期望状态收敛。控制器不需要知道"上一步做了什么"，它只关心"现在是什么状态"和"应该是什么状态"。

这个模式在 Skill 中的体现是 **Global Invariants**——一组在任何操作完成后都必须成立的不变量：

1. `openclaw --version` 返回有效版本号
2. `openclaw gateway status` 显示 RUNNING
3. 目标端口可被 curl 访问
4. 包目录中关键文件完整
5. LaunchAgent 配置指向正确路径

这些不变量就是"期望状态"的形式化描述。每次操作后，AI 必须检查所有不变量。如果有任何一条不满足，说明系统还未达到期望状态，需要继续调整——而不是宣布"完成"。

这个机制特别针对 AI 的**无状态幻觉**：AI 倾向于认为"我执行了正确的命令，所以系统现在处于正确的状态"。声明式验证强制它用事实而非假设来判断。

## 七、几个关键设计决策

除了方法论框架，Skill 中有几个具体的设计决策值得单独说。

### "两次失败必须换路"规则

```
RULE: If same approach fails twice → MUST switch strategy
NEVER: retry same npm command more than twice
```

这是从这次事故中直接提炼出的硬规则。不是建议、不是提醒，是 MUST。因为对 AI 来说，"建议"只是一个权重，在上下文足够强的时候会被覆盖。只有强制规则才能可靠地中断它的惯性。

### 信息与指令的显式区分

```
"Update available: X.Y.Z" → this is INFO, not a directive
Only user requests or explicit upgrade commands trigger upgrades
```

这个标注看起来很笨——人类不需要被告知"这是信息不是指令"。但对 AI 来说，这种显式标注是必要的认知支撑。它本质上是在给 AI 提供一个"信号分类器"，帮助它在面对模糊输入时做出正确的类型判断。

### 永远通过应用层操作

```
ALWAYS use `openclaw update` for upgrades
NEVER use `npm update/install` as first approach
```

这条规则把工具分层知识硬编码进了 Skill。AI 不需要在运行时"推理"出应该用哪层命令，因为这个知识已经被前置了。

## 八、从事件到方法论：一个更大的图景

回顾整个过程，我发现自己做的事情可以概括为三个层次的递进：

**第一层：修问题。** OpenClaw 坏了，修好它。这是最直接的价值，但也是最短暂的——下次坏了还得再修。

**第二层：修认知。** 识别出 AI 为什么会犯错，把经验编码成 Skill，让 AI 下次不会犯同样的错。这是从一次性修复到可复用知识资产的跃迁。

**第三层：修方法。** 从这个具体事件中提炼出"如何为 AI 设计认知脚手架"的方法论——状态先行、条件守卫、故障预枚举、强制策略升级。这个方法论可以迁移到任何 AI Agent 的 Skill 设计中。

这三个层次的递进，本质上对应了一个认识论的经典结构：

- 知识（Knowledge）：知道怎么修 OpenClaw
- 元知识（Meta-knowledge）：知道 AI 为什么修不好 OpenClaw
- 方法论（Methodology）：知道如何系统地预防 AI 修不好任何东西

我越来越觉得，在 AI 时代，**第二层和第三层的能力比第一层重要得多**。第一层的操作，AI 本身就在快速进步，人类的比较优势在萎缩。但识别 AI 的认知缺陷、设计补偿机制、提炼可迁移方法论——这些元认知能力，恰恰是当前 AI 最薄弱的环节，也是人类最应该发力的方向。

## 九、人机协作的认知分工

这次事件让我重新思考了人和 AI 在协作中的角色定位。

传统观点倾向于把人机协作看成"人指挥、AI 执行"的层级关系。但在实践中，它更像一个**认知互补**的关系：

- **AI 擅长**：快速执行、并行探索、不疲劳、不遗忘（在上下文窗口内）
- **AI 不擅长**：元认知判断、策略升级、信号分类、从失败中提取结构化教训
- **人类擅长**：识别模式、质疑前提、抽象概括、设计认知结构
- **人类不擅长**：深夜保持清醒、记住所有命令的 flag、不犯粗心错误

最有效的协作模式，不是人类事无巨细地指导 AI 的每一步（那还不如自己做），也不是完全放手让 AI 自由发挥（那就是我们前面看到的灾难现场）。而是**人类设计认知结构，AI 在结构内高效执行**。

Skill 就是这个认知结构的具体载体。它不是限制 AI 的牢笼，而是一副脚手架——在 AI 自身的认知能力成长到足够强大之前，帮它撑住那些它还撑不住的部分。

## 十、尾声

深夜的故障最终用了将近两个小时才完全解决。如果我一开始就自己动手，可能二十分钟就搞定了。从效率的角度看，让 AI 来修反而更慢。

但我不这么看。

那两个小时里，我获得的不只是一个修好的 Gateway，还有一套经过实战验证的 AI 认知缺陷分类体系，一个融合了四种方法论的 Skill 设计框架，以及一个关于人机协作的更清晰的认知。这些东西的保质期，比一个 Gateway 进程的 uptime 长得多。

软件工程有一句老话：**"如果你第二次手动做同一件事，就应该自动化它。"** 在 AI Agent 的语境下，我想把这句话改一改：

**"如果 AI 第二次犯同一个错，就应该给它建一个认知脚手架。"**

AI 的能力边界在快速扩展，但在任何给定的时间点，它都有认知盲区。与其等模型升级来解决这些盲区，不如我们现在就动手，用结构化的知识设计来补偿它们。这不只是在修 bug，这是在参与定义人类和 AI 协作的范式。

而这个范式的核心，也许并不是什么高深的技术，而是一个朴素的认知学原理——**好的脚手架让学习者能做到他们独自做不到的事**。无论这个学习者是人，还是一个在深夜里试图帮你修 Gateway 的 AI。
